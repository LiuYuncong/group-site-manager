## 下一步的优化计划

------

### 一、 线程安全与 GUI 响应 (并发与阻塞漏洞)

这是当前项目在 Windows 桌面端影响用户体验（导致“未响应”白屏）的核心痛点。

- **[高] 主线程密集 I/O 阻塞 (UI 层)**
  - **问题定位**：`ui/main_window.py` 的 `_process_batch_import`（批量导入数十篇文献）、`toggle_preview`（内部包含 `time.sleep(2)` 等待服务器就绪），以及 `ui/forms.py` 的所有 `_save` 方法（涉及大文件图片复制与磁盘写入），均直接在 CustomTkinter 的主事件循环中同步执行。
  - **优化建议**：必须全面实施异步化改造。将这些耗时操作包装为 `threading.Thread` 或提交至 `concurrent.futures.ThreadPoolExecutor`，并通过 `self.after(0, callback)` 将结果回传给主线程以更新 UI 状态。
- **[高] 网络 I/O 阻塞陷阱 (Core 层)**
  - **问题定位**：`core/git_engine.py` 中的 `pull_latest` 和 `commit_and_push` 包含不可控的同步网络请求。
  - **优化建议**：引擎层本身保持同步机制，但在接口的 Docstring 中必须处于最高警告级别，强制要求调用方（UI）禁止在主线程调用。
- **[中] 单例模式初始化竞态 (Core 层)**
  - **问题定位**：`core/config_manager.py` 的 `AppConfig.__new__` 单例实现未加锁。如果在应用启动瞬间被多个后台工作线程同时请求，将导致实例被重复创建。
  - **优化建议**：在类级别引入 `threading.Lock()` 确保严格的单例安全。

### 二、 跨平台兼容性与 Windows 专属陷阱

由于应用以 Windows 为主运行环境，对文件系统锁、子进程管理的容错能力需要大幅加固。

- **[高] 子进程黑框与管道死锁 (`preview_engine.py`)**
  - **问题定位 1**：使用 `pythonw.exe` 运行后，`subprocess.Popen` 调用 `hugo.exe` 会弹出原生 CMD 丑陋黑框。
  - **优化建议 1**：必须传入 Windows 特有的标志位：`creationflags=subprocess.CREATE_NO_WINDOW`。
  - **问题定位 2**：设置了 `stdout=subprocess.PIPE` 却没有线程去 `read()` 消费。当 Hugo 编译日志填满 Windows 默认的 64KB 缓冲区后，Hugo 进程将永久挂起（死锁）。
  - **优化建议 2**：将标准输出/错误重定向至 `subprocess.DEVNULL` 丢弃，或开启独立守护线程清空管道。
- **[高] 配置防损与非原子写入 (`config_manager.py`)**
  - **问题定位**：`save()` 方法直接通过 `with open(..., 'w')` 覆盖写入 `config.json`。若在写入瞬间遇到断电、蓝屏或进程强杀，配置文件将变为 0 字节。
  - **优化建议**：实施原子写入（Atomic Write）。先写入 `.tmp` 临时文件并 `flush` + `os.fsync`，成功后再使用 `os.replace` 覆盖原文件。
- **[高] 目录强制删除时的权限锁定 (`content_parser.py`)**
  - **问题定位**：`delete_item` 使用 `shutil.rmtree`。在 Windows 中，若目录下的图片正在被资源管理器预览（占用），直接删除会抛出 `PermissionError` 导致操作中断。
  - **优化建议**：为 `rmtree` 传入 `onerror` 回调函数，在失败时尝试使用 `os.chmod` 赋予可写权限后重试。
- **[中] Git 资源未释放导致 `.git/index.lock` 残留 (`git_engine.py`)**
  - **问题定位**：GitPython 的 `Repo` 实例如果不显式关闭，其底层引用的 Windows 文件句柄可能不会立即释放，阻碍其他 Git 客户端操作该仓库。
  - **优化建议**：提供显式的 `close()` 方法调用 `self.repo.close()`，并在主窗口销毁时挂载清理逻辑。

### 三、 架构设计、数据一致性与可维护性

防范数据被破坏，并为未来支持更多 Hugo 模块铺平道路。

- **[高] 手动拼接 YAML 导致的数据崩溃 (`ui/forms.py`)**
  - **问题定位**：`ProjectsTableFrame._save` 为了保存表格，手动通过字符串拼接 `---` 分隔符和缩进写入 Front-matter。这种做法极易因为特殊字符（如标题含冒号、引号）导致整个 Hugo 页面在解析时发生 Fatal Error。
  - **优化建议**：废弃手动拼接，统一使用 `frontmatter.Post(content=..., **dict)` 进行安全的序列化。
- **[高] 状态标志与实际操作不符 (`content_parser.py`)**
  - **问题定位**：`save_item` 中，如果图片源文件不存在或复制失败，函数不仅未回滚，甚至返回了 `True, "内容未保存..."`，这会欺骗 UI 层继续刷新并提示用户成功。
  - **优化建议**：严格执行失败即返回 `False`，或引入事务回滚机制（删除已创建的残缺文件夹）。
- **[中] 神级类 (God Object) 反模式 (`ui/forms.py`)**
  - **问题定位**：`ContentFormFrame` 将所有的模块 UI 逻辑（新闻、人员、论文等）全部写在一个极其庞大的类中，充满了 `if self.module_name ==` 判断，严重违背开闭原则（OCP）。
  - **优化建议**：引入策略模式 (Strategy) 或表单工厂，将 `AuthorsForm`, `PublicationForm` 拆分为独立的子类，父类只负责公共底座（如标题、日期、正文、保存/取消按钮）。
- **[中] 业务逻辑硬编码在 UI 层 (`ui/main_window.py`)**
  - **问题定位**：用于论文导入的 `AUTHOR_MAP`（课题组成员中英文对照表）被直接硬编码在了主窗口类中。
  - **优化建议**：将这类纯数据处理与映射逻辑剥离至 `core/content_parser.py`，或抽取为外部的 JSON/YAML 配置供动态读取，实现视图与数据的完全解耦。

### 四、 安全与异常边界 (Security & Edge Cases)

- **[高] Git 凭证落盘风险 (`git_engine.py`)**
  - **问题定位**：`pull` 和 `push` 失败时，捕获的 `GitCommandError.stderr` 被原样通过 `logger.error` 写入日志。若采用 HTTPS 且 URL 中夹带明文 Token，将造成严重凭证泄露。
  - **优化建议**：在写入日志前，引入正则表达式（如 `re.sub(r'://[^@]+@', '://***@', text)`）进行数据脱敏。
- **[中] 游离指针状态 (Detached HEAD) 的雪崩 (`git_engine.py`)**
  - **问题定位**：执行 `commit_and_push` 时，如果当前仓库处于游离状态，读取 `self.repo.active_branch.name` 将抛出 `TypeError` 异常，破坏无异常抛出的接口契约。
  - **优化建议**：在推送前增加 `if self.repo.head.is_detached:` 拦截，并返回友好的中文阻断提示。
- **[中] 同源文件复制崩溃 (`config_manager.py`)**
  - **问题定位**：`copy_image_as` 中，若用户选择的图片就是目标目录下的同名文件，触发 `shutil.SameFileError` 异常会导致整个保存流程中断。
  - **优化建议**：显式 `except shutil.SameFileError:`，视同复制成功并安全跳过。
